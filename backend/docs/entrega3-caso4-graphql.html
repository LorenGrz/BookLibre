Entrega 3 - Parte 2 - Caso 4

Análisis de Calificaciones con GraphQL

Este caso agrega una métrica para el tablero administrador: promedio de calificación agrupado por tipo de libro. La API GraphQL convive con la REST existente; no reemplaza ningún endpoint REST.

Qué se agregó en backend

  • Se extendió el schema GraphQL con la query analisisCalificaciones.
  • Se creó un DataFetcher DGS para resolver esa query.
  • Se creó un service específico: AnalisisCalificacionesService.
  • Se agregó una aggregation en LibroMongoRepository.
  • Se agregó un DTO de salida: PromedioCalificacionPorTipoDTO.

Query GraphQL

query {
  analisisCalificaciones {
    tipoLibro
    promedioCalificacion
  }
}

Respuesta esperada, con valores variables según los datos cargados:

{
  "data": {
    "analisisCalificaciones": [
      {
        "tipoLibro": "Coleccionable",
        "promedioCalificacion": 4.2
      },
      {
        "tipoLibro": "Comun",
        "promedioCalificacion": 3.9
      }
    ]
  }
}

De dónde sale la data

La data sale de MongoDB, colección libros. No se consulta PostgreSQL para este caso. Cada documento de libro ya tiene un campo agregado llamado calificacion, que representa el promedio calculado para ese libro.

La aggregation hace esto:

  1. Filtra libros activos con activo = true.
  2. Excluye libros sin reseñas con calificaciones.0 exists true.
  3. Agrupa por _class, que distingue LibroComun, LibroConDedicatoria y LibroColeccionable.
  4. Promedia el campo calificacion dentro de cada tipo.
  5. Mapea el nombre de clase a un tipo más simple: Comun, ConDedicatoria, Coleccionable.

Importante: este caso pide un promedio de promedios por libro. Por eso no se hace unwind de reseñas ni se promedian los valores crudos de calificaciones.valor. Cada libro calificado aporta un solo valor: su calificacion ya agregada.

Los valores devueltos no estan mockeados ni hardcodeados. Son el resultado de consultar MongoDB en el momento de ejecutar la query. Si se agregan nuevas calificaciones, cambian las calificaciones agregadas de los libros o se modifican los datos iniciales, la respuesta de analisisCalificaciones puede cambiar.

Cómo funciona GraphQL en esta entrega

GraphQL es una API de consulta. A diferencia de REST, donde cada endpoint tiene una respuesta ya definida, en GraphQL el cliente manda una query indicando exactamente que campos necesita. En esta entrega no reemplaza la API REST: convive con ella. Los endpoints REST siguen existiendo para el flujo normal de la app, y GraphQL se usa para resolver consultas de tablero administrador.

En este proyecto el endpoint GraphQL es:

POST http://localhost:8080/api/graphql

El contrato esta definido en src/main/resources/schema/schema.graphqls. Ahi se declaran las queries disponibles y los tipos de respuesta. Actualmente el schema incluye:

  • tasaConversion: caso 1, cruza clicks cacheados en Redis con reservas en PostgreSQL.
  • analisisCalificaciones: caso 4, agrupa promedios de calificacion por tipo de libro desde MongoDB.

Flujo de una consulta GraphQL:

  1. El frontend manda un POST a /api/graphql con una query.
  2. Spring Security valida la sesion. Para este endpoint se requiere rol ADMIN.
  3. Netflix DGS lee la query y busca el metodo anotado con @DgsQuery que tenga el mismo nombre.
  4. El DataFetcher no hace logica pesada: delega en un service de dominio.
  5. El service consulta la fuente de datos necesaria: Redis, PostgreSQL o MongoDB segun el KPI.
  6. DGS arma la respuesta JSON solo con los campos que pidio el cliente.

En el caso 4, la query analisisCalificaciones se resuelve con AnalisisCalificacionesDataFetcher. Ese fetcher llama a AnalisisCalificacionesService, y el service usa LibroMongoRepository para ejecutar la aggregation sobre MongoDB.

Por qué GraphQL sirve para KPIs

Un tablero de administrador suele necesitar datos de varios mundos al mismo tiempo: libros, usuarios, reservas, clicks, rankings y actividad. Con REST se podria hacer, pero muchas veces el frontend termina llamando a muchos endpoints distintos o recibiendo campos que no necesita. GraphQL permite pedir varias metricas en una sola request, por ejemplo: 

query {
  tasaConversion {
    titulo
    clicks
    reservas
    tasaConversion
  }
  analisisCalificaciones {
    tipoLibro
    promedioCalificacion
  }
}

DGS orquesta esa consulta: llama a los resolvers necesarios y cada resolver delega en su service. Asi el cliente puede armar un tablero flexible sin conocer todos los endpoints internos ni mezclar la logica de calculo en el front.

Cómo funciona Redis en esta entrega

Redis se usa como cache en memoria para datos muy consultados o costosos de calcular. La idea central es evitar recalcular informacion pesada en cada request. En lugar de ir siempre a MongoDB o PostgreSQL, se guarda un resultado preparado en Redis y se lee desde ahi cuando la pantalla lo necesita.

En la Parte 1, el home usa el cache de libros populares. El objeto cacheado se guarda bajo la entidad TopLibrosCache, con key HOME, dentro del hash Redis top_libros. Ese cache contiene una lista de libros populares y el total de libros activos.

Flujo del cache de libros populares:

  1. Cuando el home necesita populares, LibroService.getLibrosPopulares() intenta leer Redis.
  2. Si Redis tiene la key HOME, devuelve esos libros sin recalcular el ranking.
  3. Si hay cache miss, LibroService.actualizarCacheTopLibros() consulta MongoDB y guarda el resultado en Redis.
  4. Cuando un usuario entra al detalle de un libro, ClickLogService.registrarClick() registra el click en MongoDB.
  5. Si el click fue valido, se vuelve a actualizar el cache de top libros para que el home refleje el nuevo ranking.

Redis no reemplaza a MongoDB ni a PostgreSQL. Redis guarda una vista rapida y temporal de datos ya calculados. MongoDB sigue siendo la fuente de verdad para libros, clicks y calificaciones documentales; PostgreSQL sigue siendo la fuente de verdad para usuarios, reservas y datos relacionales.

Redis y GraphQL juntos

Redis tambien puede participar en KPIs GraphQL. El ejemplo directo de esta rama es tasaConversion. Esa query toma el top 5 de libros mas clickeados desde Redis y cruza esos libros con PostgreSQL para contar reservas. Con esos dos datos calcula: 

tasaConversion = reservas / clicks

Esto muestra la idea de la entrega: GraphQL actua como capa de orquestacion y cada service decide de donde trae la informacion. Un KPI puede leer solo MongoDB, como analisisCalificaciones; otro puede combinar Redis y PostgreSQL, como tasaConversion.

Resumen para defender

  • REST sigue funcionando para la aplicacion normal.
  • GraphQL agrega una API flexible para el tablero administrador.
  • El schema define que se puede consultar y que forma tiene cada respuesta.
  • Los DataFetcher DGS conectan cada query con un service.
  • Los services mantienen la logica de negocio y eligen la fuente de datos correcta.
  • Redis acelera lecturas frecuentes, especialmente el top de populares del home.
  • MongoDB mantiene libros, clicks y calificaciones; PostgreSQL mantiene reservas y usuarios.
  • El caso 4 no usa Redis porque su metrica sale directamente de calificaciones agregadas en MongoDB.

Cómo levantar el backend local

Desde el worktree o repo backend:

cd /Users/dantestefanotripodi/.config/superpowers/worktrees/backend-2026-grupo-12/caso4-graphql-backend

Levantar dependencias locales:

docker compose up -d redis postgres configsvr1 configsvr2 configsvr3 shard1a shard1b shard1c shard2a shard2b shard2c mongo-rs-init mongos-router1 mongos-router2 mongo

Cuando Mongo termine la inicialización, correr la app Spring Boot:

./gradlew bootRun

El backend queda disponible en:

http://localhost:8080

Cómo probar GraphQL con curl

El endpoint /api/graphql requiere rol admin. El usuario seed de admin es admin@booklibre.com con contraseña admin123.

Primero hacer login y guardar cookies:

curl -i -c /tmp/booklibre-cookies.txt \
  -H 'Content-Type: application/json' \
  -d '{"email":"admin@booklibre.com","password":"admin123"}' \
  http://localhost:8080/api/login

Después consultar GraphQL usando esas cookies:

curl -s -b /tmp/booklibre-cookies.txt \
  -H 'Content-Type: application/json' \
  -d '{"query":"query { analisisCalificaciones { tipoLibro promedioCalificacion } }"}' \
  http://localhost:8080/api/graphql

Cómo probar desde el frontend

En otra terminal, levantar el frontend desde la rama del caso 4:

cd /Users/dantestefanotripodi/.config/superpowers/worktrees/frontend-2026-grupo-12/caso4-graphql-frontend
pnpm dev

Abrir:

http://localhost:5173/consultas

Loguearse como admin si hace falta, expandir la tarjeta Análisis de Calificaciones [GraphQL] y hacer click en Consultar.

Qué hace el frontend

  • adminService.obtenerAnalisisCalificaciones() manda un POST GraphQL a /graphql.
  • ConsultasAdmin guarda resultado, loading y error en estado local.
  • La tarjeta renderiza el tipo de libro con label humano usando TIPO_LABEL.
  • El promedio se muestra con dos decimales usando toFixed(2).
